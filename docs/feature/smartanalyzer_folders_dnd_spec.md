# SmartAnalyzer — Folders + Drag & Drop

Евгений, ниже единый документ для Cursor/Codex: **спека + чеклист + готовый промпт** на реализацию фичи папок, системных папок по инструментам и drag & drop для SmartAnalyzer.

---

## 1. Цель фичи

Сейчас дашборд SmartAnalyzer в основном выполняет роль истории запусков. Нужно превратить его в **рабочее пространство для постоянной работы с документами и результатами анализов**.

Фича должна дать пользователю:

- создавать собственные папки;
- видеть системные папки по инструментам;
- загружать файлы сразу в нужную папку;
- перемещать документы и результаты анализов между папками;
- использовать drag & drop как основной быстрый сценарий;
- быстрее возвращаться к нужным документам и анализам.

Главная продуктовая идея:

> SmartAnalyzer должен выглядеть не как список прошлых запусков, а как рабочий кабинет для управления документами, анализами и проектами.

---

## 2. Продуктовая гипотеза

Если добавить папки, системную группировку по инструментам и drag & drop, то:

- пользователю станет проще ориентироваться в истории;
- вырастет retention, потому что в продукт захочется возвращаться;
- продукт будет восприниматься как полноценная рабочая система, а не разовый AI-инструмент;
- станет проще масштабировать продукт в B2B и командную работу.

---

## 3. Scope MVP

В MVP входят:

1. Пользовательские папки.
2. Системные папки по инструментам.
3. Привязка документа к папке.
4. Привязка анализа к папке.
5. Drag & drop перемещение документов и анализов между папками.
6. Drag & drop upload файла в выбранную папку.
7. Список папок в левой колонке.
8. Просмотр содержимого выбранной папки в центре экрана.
9. Базовые действия: создать, переименовать, удалить папку.
10. Фильтрация и поиск внутри текущей папки.

Не входят в MVP:

- вложенные папки;
- shared/team folders;
- granular permissions;
- comments;
- smart folders с динамическими правилами;
- bulk drag для десятков элементов со сложной логикой;
- version tree документа.

---

## 4. Основные сущности

Нужно явно разделить 3 сущности:

### 4.1 Folder
Папка — контейнер для документов и/или анализов.

Типы папок:
- `system` — системная папка по инструменту;
- `user` — созданная пользователем папка.

### 4.2 Document
Исходный файл пользователя:
- pdf
- docx
- txt
- другие поддерживаемые форматы

### 4.3 Analysis
Результат запуска конкретного инструмента по документу.

Примеры:
- document-analyzer
- contract-checker
- data-extractor
- tender-analyzer
- risk-analyzer

Важно:
- один документ может иметь много анализов;
- документ и анализ должны быть логически связаны;
- пользователь должен иметь возможность открыть и сам документ, и конкретный анализ.

## 5. Продуктовая логика

### 5.1 Системные папки
При открытии раздела пользователь видит системные папки:

- Все документы
- Недавние
- Избранное (можно заложить под будущее, UI можно скрыть до реализации)
- По инструментам:
  - Анализатор документов
  - Проверка рисков
  - Извлечение данных
  - Tender Analyzer
  - Risk Analyzer

Минимум для MVP:
- `Все документы`
- по одной системной папке на каждый инструмент

Системные папки нельзя удалить и переименовать.

### 5.2 Пользовательские папки
Пользователь может создавать свои папки, например:
- Клиент A
- Аренда
- Судебка
- Поставщики
- Срочно проверить

Пользовательские папки:
- можно создавать;
- можно переименовывать;
- можно удалять;
- нельзя делать вложенными в MVP.

### 5.3 Куда попадает новый файл
Сценарии:

#### Сценарий A — пользователь загрузил файл из конкретной папки
Файл сохраняется в выбранную папку.

#### Сценарий B — пользователь загрузил файл из общего экрана
Файл сохраняется в дефолтную пользовательскую область `Все документы`.

#### Сценарий C — пользователь запускает инструмент напрямую
Документ и его анализ:
- попадают в текущую выбранную папку, если она выбрана;
- иначе попадают в `Все документы`.

### 5.4 Как отображаются анализы в системных папках
Системная папка по инструменту показывает **анализы данного типа**.

Пример:
- папка `Risk Analyzer` показывает только результаты risk-analyzer;
- папка `Document Analyzer` показывает только результаты document-analyzer.

### 5.5 Перемещение
Пользователь может перемещать:
- документ;
- анализ;
- документ вместе с анализами — не в MVP по drag как связка, только отдельно или через будущий bulk action.

Для MVP проще:
- документ перемещается как отдельная сущность;
- анализ перемещается как отдельная сущность;
- если анализ привязан к документу, это не ломает связь.

## 6. UX / UI требования

### 6.1 Общая структура экрана

- По инструментам
  - Анализатор документов
  - Проверка рисков
  - Извлечение данных

  - Risk Analyzer
- Мои папки
  - список пользовательских папок
- Кнопка `+ Новая папка`

#### Центральная область
Содержимое выбранной папки:

Режим отображения MVP:
- таблица

Колонки:
- Тип (`document` / `analysis`)
- Инструмент
- Название
- Дата
- Статус
- Действия

Дополнительно допустимо:
- число рисков
- размер файла

### 6.2 Статусы
Для analysis:
- `queued`
- `processing`
- `completed`
- `failed`

Для document можно показывать:
- `uploaded`

### 6.3 Drag & Drop сценарии

#### DnD-1: Перетаскивание элемента в папку
Пользователь перетаскивает строку документа или анализа на папку слева.

Ожидаемое поведение:
- целевая папка подсвечивается;
- на drop вызывается перемещение;
- список текущей папки обновляется;
- появляется toast `Элемент перемещён`.

#### DnD-2: Upload файла в текущую папку
Пользователь кидает файл:
- либо в центральную область;
- либо на папку слева.

Ожидаемое поведение:
- показывается зона загрузки;
- файл уходит на upload;
- после успешной загрузки появляется документ в папке.

#### DnD-3: Drag over visual feedback
При drag over:
- папка подсвечивается;
- центральная область показывает `Отпустите файл для загрузки` или `Отпустите для перемещения`.

### 6.4 Context menu / row actions
Для пользовательских папок:
- Переименовать
- Удалить

Для документа:
- Открыть
- Скачать
- Переместить
- Удалить
- Открыть инструмент

Для анализа:
- Подробнее
- Скачать JSON
- Переместить
- Открыть инструмент
- Удалить

## 7. Предлагаемая модель БД

Ниже минимальная схема без лишней сложности.

### 7.1 Таблица folders

Поля:
- `id` uuid pk
- `user_id` uuid not null
- `name` varchar not null
- `slug` varchar null
- `type` varchar not null (`system` | `user`)
- `system_key` varchar null
- `created_at` timestamptz not null
- `updated_at` timestamptz not null
- `deleted_at` timestamptz null

Правила:
- системные папки создаются автоматически;
- для `system` папок `system_key` обязателен;
- для `user` папок `system_key` null.

Примеры `system_key`:
- `all_documents`
- `tool_document_analyzer`
- `tool_contract_checker`
- `tool_data_extractor`
- `tool_tender_analyzer`
- `tool_risk_analyzer`

### 7.2 Изменения в documents

Добавить:
- `folder_id` uuid null

Поля documents уже есть в проекте:
- `id`
- `user_id`
- `filename`
- `file_path`
- `mime_type`
- `size_bytes`
- `created_at`

Рекомендуемое поведение:
- для пользовательской логики хранить `folder_id`;
- если `folder_id` null, считать документ лежащим в `all_documents`.

Лучше сразу хранить `folder_id`, чтобы не плодить условности.

### 7.3 Изменения в document_analyses

Добавить:
- `folder_id` uuid null
- `status` varchar not null default `completed`

У тебя уже есть:
- `id`
- `document_id`
- `user_id`
- `tool_slug`
- `result_json`
- `created_at`

Итог:
- анализ знает папку;
- анализ знает инструмент;
- анализ знает статус.

### 7.4 Индексы

Нужны индексы:
- `folders(user_id, type)`
- `folders(user_id, system_key)` unique where deleted_at is null
- `documents(user_id, folder_id, created_at desc)`
- `document_analyses(user_id, folder_id, created_at desc)`
- `document_analyses(user_id, tool_slug, created_at desc)`

## 8. API контракт

Префикс как в проекте: `/api/v1`

Ошибка как в проекте:

```json
{
  "error": "SOME_ERROR_CODE",
  "message": "Human readable message",
  "details": {}
}
```

### 8.1 Получить папки
`GET /api/v1/folders`

Response 200:

```json
{
  "items": [
    {
      "id": "uuid",
      "name": "Все документы",
      "type": "system",
      "system_key": "all_documents",
      "item_count": 12,
      "created_at": "2026-03-08T10:00:00Z"
    }
  ]
}
```

### 8.2 Создать папку
`POST /api/v1/folders`

Request:

```json
{
  "name": "Клиент A"
}
```

Response 201:

```json
{
  "id": "uuid",
  "name": "Клиент A",
  "type": "user",
  "system_key": null,
  "created_at": "2026-03-08T10:00:00Z"
}
```

### 8.3 Переименовать папку
`PATCH /api/v1/folders/{id}`

Request:

```json
{
  "name": "Клиент A / договоры"
}
```

### 8.4 Удалить папку
`DELETE /api/v1/folders/{id}`

Поведение:
- удалять можно только `user` папку;
- если в папке есть элементы, для MVP можно сделать мягкое поведение:
  - документы и анализы переносятся в `all_documents`;
  - либо запрет на удаление непустой папки.

Рекомендация для MVP:
- **запретить удаление непустой папки**, чтобы логика была проще.

### 8.5 Получить содержимое папки
`GET /api/v1/folders/{id}/items`

Query params:
- `q`
- `type=document|analysis`
- `status`
- `page`
- `page_size`

Response 200:

```json
{
  "items": [
    {
      "id": "uuid",
      "entity_type": "analysis",
      "title": "fake_contract_risky_sample.docx",
      "tool_slug": "document-analyzer",
      "status": "completed",
      "created_at": "2026-03-06T11:00:00Z",
      "document_id": "uuid"
    },
    {
      "id": "uuid",
      "entity_type": "document",
      "title": "supplier_contract_v2.pdf",
      "mime_type": "application/pdf",
      "created_at": "2026-03-06T11:05:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "page_size": 20,
    "total": 52
  }
}
```

### 8.6 Переместить документ
`POST /api/v1/documents/{id}/move`

Request:

```json
{
  "folder_id": "uuid"
}
```

### 8.7 Переместить анализ
`POST /api/v1/analyses/{id}/move`

Request:

```json
{
  "folder_id": "uuid"
}
```

### 8.8 Upload документа в папку
`POST /api/v1/documents/upload`

Поддержать опциональный параметр:
- `folder_id`

Multipart form-data:
- `file`
- `folder_id` optional

### 8.9 Запуск инструмента в папке
`POST /api/v1/tools/{tool-slug}/run`

Нужно расширить контракт, чтобы можно было передавать `folder_id`.

Пример:

```json
{
  "document_id": "uuid",
  "folder_id": "uuid",
  "options": {}
}
```

## 9. Backend implementation notes

### 9.1 Автосоздание системных папок
При первом входе пользователя в раздел дашборда или при регистрации нужно гарантировать создание системных папок.

Можно сделать helper/service:
- `ensure_user_system_folders(user_id)`

Вызывается:
- после регистрации;
- перед выдачей `GET /folders`.

### 9.2 Folder service
Нужен отдельный сервисный слой:
- create folder
- rename folder
- delete folder
- list folders
- list folder items
- move document
- move analysis
- ensure system folders

### 9.3 Системные папки не должны зависеть только от folder_id
Например `Все документы` — это скорее виртуальная папка.

Рекомендуемая реализация для MVP:
- `all_documents` возвращает все документы и анализы пользователя независимо от `folder_id`;
- папки по инструментам возвращают анализы по `tool_slug`;
- user folders возвращают элементы по `folder_id`.

То есть некоторые системные папки — **виртуальные представления**, а не физическое единственное место хранения.

Это важный архитектурный момент.

### 9.4 Move в системные папки
Рекомендуемое поведение:
- в `all_documents` перемещать нельзя как в физическую папку;
- в tool-папки тоже физически перемещать не нужно;
- физически перемещать можно только в `user` папки.

Системные папки — представления/фильтры.

Это упрощает модель.

- loading states;

### 10.3 Drag & Drop library
Рекомендуется взять что-то современное и простое.

Если нужен минимальный путь — можно использовать нативный HTML5 drag and drop.
Если нужен более стабильный DX/UI — взять библиотеку уровня dnd-kit.

Рекомендация:
- для MVP можно использовать **dnd-kit**, если команда ок с зависимостью;
- иначе нативный drag/drop тоже допустим.

### 10.4 UX corner cases
Продумать:
- нельзя бросать папку в саму себя;
- нельзя бросать элемент в ту же папку;
- если move не удался — откатить optimistic update;
- если папка системная и не принимает move — не показывать её как drop target для физического перемещения.

### 10.5 Empty states
Нужны состояния:
- у пользователя ещё нет папок;
- в папке пока пусто;
- поиск внутри папки ничего не нашёл;
- ошибка загрузки;
- ошибка перемещения.

## 11. Security / access rules

Так как продукт уже multi-user, нужно жёстко проверять ownership.

На backend любой запрос к папке, документу или анализу должен проверять:
- сущность принадлежит текущему пользователю;
- пользователь не может двигать чужие документы;
- пользователь не может переименовать/удалить системные папки.

Ошибки:
- `FOLDER_NOT_FOUND`
- `FOLDER_ACCESS_DENIED`
- `SYSTEM_FOLDER_MUTATION_FORBIDDEN`
- `INVALID_MOVE_TARGET`
- `DOCUMENT_NOT_FOUND`
- `ANALYSIS_NOT_FOUND`

## 13. Acceptance criteria

Фича считается принятой, если:

1. Пользователь видит слева системные папки и свои папки.
2. Пользователь может создать новую папку.
3. Пользователь может переименовать свою папку.
4. Пользователь может удалить пустую свою папку.
5. Пользователь может открыть папку и увидеть её содержимое.
6. Пользователь может загрузить файл сразу в выбранную папку.
7. Пользователь может перетащить документ в пользовательскую папку.
8. Пользователь может перетащить анализ в пользовательскую папку.
9. После обновления страницы структура папок и содержимое сохраняются.
10. Системные папки нельзя удалить и переименовать.
11. Переход в системную папку по инструменту показывает только соответствующие анализы.
12. Ошибки перемещения и загрузки корректно показываются пользователю.
